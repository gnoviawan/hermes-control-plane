# Hermes Control Plane Phase 1 Implementation Plan

> For Hermes: execute autonomously in small verified steps, commit frequently, open GitHub issues for any blocker/defect, and keep Phase 1 shippable at all times.

**Goal:** Build and deploy a QwenPaw-inspired Hermes dashboard in a new repository without modifying Hermes source code, using a separate Hermes sidecar service so the current production Hermes keeps running.

**Architecture:** A React + Ant Design frontend provides the QwenPaw-like shell and management pages. A FastAPI adapter backend reads Hermes state from `HERMES_HOME`, invokes Hermes CLI where needed, and exposes stable REST endpoints for the frontend. A separate Hermes sidecar container runs alongside the dashboard for non-destructive Hermes-backed operations and future expansion.

**Tech Stack:** React 18, TypeScript, Vite, Ant Design, Zustand, React Router, FastAPI, Uvicorn, Python 3.13, Docker Compose, Dokploy.

---

## Phase 0 — Discovery and constraints
- Verify GitHub auth, local toolchain, Hermes CLI capabilities, and current Dokploy access state.
- Reuse only public Hermes APIs / CLI / filesystem; do not patch Hermes source.
- Record Dokploy auth limitation if deployment API/browser access is unavailable.

## Phase 1 — Shippable product scope
1. QwenPaw-like shell
   - Header + grouped sidebar
   - Agent-scoped vs global sections
   - Persistent selected profile state
2. Profile management
   - List profiles
   - Show active profile, paths, gateway state
   - Create profile (clone / clone-all options)
3. Skill management
   - List skills and enabled status per profile
   - Toggle skill enable/disable for active profile
   - Broadcast skill configuration from one profile to one or many target profiles
4. Operations pages
   - Status overview
   - Sessions explorer
   - Cron jobs
   - Logs
5. Deployment package
   - Separate `dashboard` service
   - Separate `hermes-sidecar` service
   - Host mounts compatible with `/opt/data` and `/opt/hermes`
   - Health checks and documented Dokploy setup

## Phase 2 — Verification gates
- Local frontend build passes
- Backend import/lint/smoke tests pass
- API smoke tests pass
- GitHub PR opened, CI green, merged to `main`
- Dokploy deployment healthy
- Public URL verified manually and by HTTP checks
- Any defect found during CI/deploy/prod verification gets a GitHub issue and fix commit

## Initial execution sequence
1. Scaffold repo structure and tooling
2. Implement backend adapter routes
3. Implement frontend shell + core pages
4. Add tests and CI
5. Run local verification
6. Open PR and fix CI until green
7. Merge
8. Deploy to Dokploy with separate services
9. Verify URL + logs and iterate until green
